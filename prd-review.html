PRD v3 方案评审与建议

基于"价格视角"定位对 Deals × Family Fusion PRD v3 的分析，识别方向风险并提供调整建议

!

核心矛盾：方案在两个方向之间摇摆

PRD v3 实质上在做什么

PRD v3 构建了一个 六层页面架构（L1-L6）× 人群矩阵（5段）× 旅行阶段（7期）= 超级个性化推荐频道。

它的核心驱动力是"比首页更精准地推荐"——按人群分（家庭/商旅/学生/钻石会员/黑盒）、按阶段分（灵感→比价→选品→时机→打包→行中→行后）。

这恰好是定位指南中标注的最大风险：做成"个性化推荐 + 划线价"，与首页/行程页无本质区别，频道失去独立存在的理由。

PRD v3 的逻辑

用户是谁 × 用户在哪个阶段 → 推什么货

核心竞争力 = 个性化精准度
与首页/行程页的区别 = 更多人群颗粒度

定位指南的逻辑

发生了什么价格事件 → 哪些用户受益

核心竞争力 = 价格判断的可信度
与首页/行程页的区别 = 视角完全不同

判断标准：如果把所有价格标签去掉，PRD v3 的页面和首页推荐有什么区别？如果答案是"更精准的推荐"——那方向可能错了。Deals 应该提供的是"首页不可能提供的东西"，而不是"首页的升级版"。
1

L2 "黑盒主战场"：方向对了，执行需修正

PRD 的做法

L2 定位为"实时 engagement slot"，服务 58% 的无标签黑盒用户，做实时推荐。

建议调整

58% 无标签用户反而是好事——对这些用户不需要猜他们是谁。展示的应该是客观的价格事件："什么东西正在变便宜"，而不是"你可能喜欢什么"。

L2 应该做成"今日最强价格信号"：

  • 当天降幅最大的 3-5 个 Deal（跨品类）
  • 即将到期的限时促销（紧迫性最高）
  • 刚突破历史低价的商品（新鲜感最强）

这些信息对任何人都有价值，不依赖用户画像。"东京机票今天比上周低 40%"这件事，不管你是家庭用户还是商旅用户，都是有效信息。

2

六层架构过重，建议收窄为三层

问题

6 层 × 5 人群 × 7 阶段 = 需要大量内容填充矩阵。为了不留空白，必然引入第三梯队（弱信号）商品。结果：频道变成"什么都有但没什么特别便宜的"。

PRD 自己的数据也印证了这一点：L6 的 Flash Deals + IP 内容点击率仅 4.2%，说明层级太深用户已经不看了。

建议：3 层核心架构

┌─────────────────────────────────────────────────────────┐
│ LAYER 1: 闪购层（Flash） — 限时 + 强价格事件
│  • 倒计时驱动
│  • 仅放第一梯队供给（航司闪促/历史最低/平台补贴）
│  • 用户价值："再不买就没了"
├─────────────────────────────────────────────────────────┤
│ LAYER 2: 发现层（Discover） — 按品类/目的地的价格洼地
│  • 当季低价目的地 / 品类最值得买
│  • 放第一、第二梯队供给
│  • 用户价值："原来现在去这里最划算"
├─────────────────────────────────────────────────────────┤
│ LAYER 3: 个人层（For You） — 你关注的东西降价了
│  • 基于搜索/收藏/历史浏览的降价提醒
│  • 唯一依赖用户画像的层（但驱动力仍是"降价"）
│  • 用户价值："我盯的那个终于降了"
└─────────────────────────────────────────────────────────┘

L1 标题层 + L4 矩阵层 + L5 商品流 + L6 IP内容
→ 合并或降级为辅助模块，不作为独立层级
人群标签（家庭/商旅/学生）不应该重建整个页面，而是用来调整三层内的排序权重。家庭用户看到的"发现层"里亲子酒店排前面，但页面结构和逻辑完全一样。
3

"人群×阶段"矩阵的运营成本被严重低估

矩阵的真实成本

维度 数量 问题
人群分类 5 种 标签覆盖率低（58%无标签）
旅行阶段 7 期 阶段判断准确率未知
矩阵单元格 35 种状态 每种需要独立内容策略
实际可维护性 — 运营团队不可能持续管理 35 条内容线

建议：用一个简单二分法替代

路径 A：有明确行程

"你的行程相关降价"

检测到用户有搜索/收藏/订单 → 展示相关商品的价格变动

路径 B：无明确行程

"当前市场上最值得买的东西"

纯价格事件驱动 → 展示最强折扣信号

两条路径覆盖 100% 用户。实现成本是矩阵方案的 1/10，但用户感知差异不大——因为真正影响决策的是"价格证据"而不是"页面是否知道我有小孩"。

4

TripGen AI 助手：功能对但定位偏

PRD 的做法

悬浮 AI 按钮（FAB），功能是对话式组装行程——"帮我规划东京 5 天行程"。

问题

用户来 Deals 的心理是"找便宜"，不是"规划行程"。一个行程规划 AI 在 Deals 里会：

  • 分散用户注意力（从"扫货"变成"聊天"）
  • 模糊频道心智（这到底是优惠频道还是规划工具？）
  • 与其他场域（行程页）功能重叠

建议：让 AI 服务于"价格决策"

如果保留 AI，让它回答价格问题而非规划问题：

用户问题 AI 回答方向
"7月去东京什么时候最便宜？" 展示价格日历 + 推荐最低价日期
"这个价格是近期最低吗？" 对比近30/60/90天价格走势
"有没有比这更划算的？" 展示同目的地/同类型的更低价替代
"3000 预算能去哪？" 按预算筛选当前有价格优势的目的地
"机酒分开买还是一起买划算？" 计算分开 vs 打包的价差

本质：AI 是价格顾问（"我帮你算这个划不划算"），不是行程规划师（"我帮你安排去哪玩"）。后者放在行程页更合适。

5

事件驱动模块（演唱会/F1）：方向正确，传达需调整

为什么这个模块方向正确

PRD 数据：演唱会/F1 联动占 HK 流量 15%。这是天然的"价格时机"场景：

  • 用户有明确时间窗口（演出日期固定）
  • 周边酒店/机票价格波动剧烈（事件溢价 100-300%）
  • 用户有强烈的"找到平价选项"需求

传达方式建议

当前传达

"The Weeknd 曼谷演唱会套餐"

¥3,299 · 机+酒+门票

→ 像一个旅游产品广告

建议传达

"曼谷 3/15 周边酒店比平日贵 180%"

但这 3 家还在平价区间 ¥450-680/晚

→ 像一个帮你避坑的价格顾问

区别：前者是"卖产品"，后者是"帮你省钱"。后者才符合 Deals 的价格视角定位。

6

"老客通道"：不需要开关，自动切换即可

PRD 的做法

一个手动"老客通道开关"，让忠实用户简化页面、隐藏新客促销。

建议：根据访问频次自动切换

用户类型 判断条件 页面表现
首次/低频 近30天访问 ≤ 2次 完整教育版：强调"这里为什么值得逛"+ 频道价值说明 + 信任声明
中频 近30天访问 3-10次 标准版：三层架构完整展示
高频 近30天访问 > 10次 极简版：今日新增 Deals 列表 + 你关注的降价，跳过所有包装

让用户手动切换是多一个认知成本。系统应该默认做对的事。

∑

总结：六条调整建议对照表

维度 PRD v3 当前方向 建议调整方向
选品驱动力 人群画像驱动 价格事件驱动 + 人群排序
页面结构 6 层 × 35 态矩阵（重） 3 层核心 + 2 路径（轻）
差异化来源 比首页更精准的推荐 首页不提供的价格视角
AI 功能 行程规划助手 价格决策顾问
填充策略 矩阵需要大量供给填满 宁少勿滥，保浓度
成功标准 推荐精准度（CTR） 价格信任度 + 回访率

一句话判断

PRD v3 在把 Deals 做成一个"更聪明的推荐频道"——但 Deals 真正的机会是做
"Trip.com 里唯一提供价格视角的地方"。

建议在 v3 框架基础上，把核心驱动力从"人群精准度"切换为"价格事件强度"。

✓

PRD v3 中值得保留的部分

事件驱动模块

演唱会/F1 联动有真实的价格波动场景，用户价值清晰。保留并强化价格视角的传达方式。

降价提醒机制

PRD 中"你搜过的东西降价了"的通知逻辑完全正确——这是"个人层"的核心功能。

Lock Price 机制

¥98 锁价功能天然适合 Deals 场景——"现在是好价但还没确定行程"的用户有真实需求。

跨城火车连接

香港→广州、曼谷→清迈的跨城推荐，如果叠加"这条线今天最低 ¥XX"就是好的 Deal 内容。
