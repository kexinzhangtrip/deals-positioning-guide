A2UI 协议核心思路

A2UI (Agent-to-UI) 是 Google 提出的声明式 UI 协议，让 AI Agent 通过 JSON 描述 UI 意图，客户端用原生组件渲染。核心理念：LLM 生成结构化描述，不生成代码。

为什么要分层？

安全性

声明式 JSON 不可执行，Agent 只能选取预定义组件，防注入攻击。

LLM 友好

扁平结构比深层嵌套更适合 LLM 生成——减少括号匹配错误。

渐进式渲染

JSONL 流式输出，生成一个组件就渲染一个，不等完整树。

跨平台

同一份描述 → React / Flutter / 小程序用各自原生控件渲染。

❌ 传统方式

LLM 生成 HTML/CSS/JS

不安全、不可控、跨平台差

✓ A2UI 方式

LLM 生成组件描述 JSON

安全、受控、平台无关

五层分离总览

┌─────────────────────────────────────────────────────────────┐
│ Layer 5: Theme — 视觉样式（颜色/间距/字体/圆角）
├─────────────────────────────────────────────────────────────┤
│ Layer 4: Action — 交互行为（点击/提交/滚动 → 回调 Agent）
├─────────────────────────────────────────────────────────────┤
│ Layer 3: Catalog — 可用组件约束（Agent 只能用这些类型）
├─────────────────────────────────────────────────────────────┤
│ Layer 2: Data Model — 动态数据绑定（值变了 UI 自动更新）
├─────────────────────────────────────────────────────────────┤
│ Layer 1: Structure — 组件树骨架（扁平邻接表 + ID 引用）
└─────────────────────────────────────────────────────────────┘

↕ 各层独立变更，互不影响
结构不变时换主题 → 只改 Theme
数据刷新时 → 只发 Data Model update
新增组件类型 → 只扩展 Catalog
层级职责变更频率训练目标
Structure定义"有哪些组件、怎么嵌套"页面级LLM 学会布局选择
Data Model定义"数据从哪来、绑定到哪"实时LLM 学会数据视图分离
Catalog定义"能用哪些组件类型"版本级LLM 只在边界内生成
Action定义"用户操作触发什么"交互级LLM 学会声明式行为
Theme定义"长什么样"品牌级LLM 学会样式与结构解耦

Layer 1: 结构层 (Structure)

定义组件树的骨架和父子关系。采用扁平邻接表模型——所有组件平铺在数组里，通过 ID 引用建立层级。

核心字段

字段类型说明
idstring组件唯一标识，全局不重复
componentstring组件类型，必须在 Catalog 中存在
childrenstring[]子组件 ID 列表（容器类组件）
childstring单子组件 ID（单槽组件如 Card）

示例

Structure JSON {
  "updateComponents": {
    "surfaceId": "deals-page",
    "components": [
      { "id": "root", "component": "Column", "children": ["header", "card-1", "card-2"] },
      { "id": "header", "component": "Text", "text": "今日闪购" },
      { "id": "card-1", "component": "Card", "children": ["img-1", "info-1"] },
      { "id": "img-1", "component": "Image", "url": "..." },
      { "id": "info-1", "component": "Column", "children": ["title-1", "price-1"] },
      { "id": "title-1", "component": "Text", "text": "东京新宿希尔顿" },
      { "id": "price-1", "component": "PriceTag" }
    ]
  }
}
关键设计：扁平列表而非嵌套树。LLM 生成时无需跟踪括号深度，每个组件独立一行，流式输出时可逐个渲染。

Layer 2: 数据层 (Data Model)

将动态数据与 UI 结构完全解耦。组件通过 JSON Pointer 绑定数据源，数据变化时只需发送 updateDataModel，无需重新生成组件树。

数据模型定义

Data Model {
  "updateDataModel": {
    "surfaceId": "deals-page",
    "value": {
      "hotels": [
        {
          "id": "htl_001",
          "name": "东京新宿希尔顿",
          "currentPrice": 599,
          "originalPrice": 890,
          "discount": "-35%",
          "rating": 4.7,
          "remainingRooms": 3
        }
      ]
    }
  }
}

组件绑定数据

Binding // 组件通过 binding 字段引用数据模型中的路径
{ "id": "price-1", "component": "PriceTag",
  "bindings": {
    "current": "/hotels/0/currentPrice",
    "original": "/hotels/0/originalPrice",
    "badge": "/hotels/0/discount"
  }
}

增量更新

Partial Update // 价格变了，只更新数据，UI 自动响应
{
  "updateDataModel": {
    "surfaceId": "deals-page",
    "path": "/hotels/0/currentPrice",
    "value": 549
  }
}
好处：Agent 回答"价格查询"时只需更新 Data Model，不用重新描述整个 UI 结构。训练时 LLM 学会区分"改结构"和"改数据"。

Layer 3: 组件目录层 (Catalog)

Catalog 定义了 LLM 能使用的所有组件类型及其合法属性。这是安全边界——Agent 无法创造 Catalog 之外的组件。相当于给 LLM 一个"菜单"。

Catalog Schema 示例

Component Catalog {
  "catalogId": "deals-v1",
  "components": {

    "Column": {
      "description": "纵向容器",
      "props": { "children": "string[]", "gap": "number", "alignment": "start|center|end" }
    },

    "Card": {
      "description": "卡片容器，带圆角和阴影",
      "props": { "children": "string[]", "elevation": "number", "padding": "number" }
    },

    "Text": {
      "description": "文本组件",
      "props": { "text": "string", "variant": "h1|h2|body|caption", "binding": "jsonPointer" }
    },

    "PriceTag": {
      "description": "价格展示，含现价/原价/折扣标签",
      "props": { "bindings": "object", "currency": "string", "showOriginal": "boolean" }
    },

    "Button": {
      "description": "可点击按钮",
      "props": { "label": "string", "variant": "primary|secondary|text", "action": "Action" }
    },

    "Countdown": {
      "description": "倒计时组件",
      "props": { "endTime": "ISO8601", "label": "string" }
    },

    "Image": {
      "description": "图片组件",
      "props": { "url": "string", "alt": "string", "aspectRatio": "string" }
    },

    "Badge": {
      "description": "标签/徽章",
      "props": { "text": "string", "color": "success|warning|error|info" }
    }

  }
}

训练意义

Catalog 放在 system prompt 中，LLM 被约束为"只能用菜单上的菜"。训练时 pair 的格式是：(需求描述, Catalog) → 合法组件组合。如果 LLM 生成了 Catalog 中不存在的 component 值，validation 直接拒绝。

Layer 4: 交互层 (Action)

定义用户操作如何声明式地回调 Agent——不是"执行代码"，而是"发送事件描述"。Agent 收到事件后决定下一步响应。

Action 类型

类型触发时机用途
callback用户点击/提交发送事件给 Agent，等待响应
navigate用户点击跳转到指定 URL 或页面
updateLocal用户交互仅更新本地 Data Model，不回调
dismiss用户关闭移除当前 surface

示例：预订按钮回调

Action Definition {
  "id": "book-btn",
  "component": "Button",
  "label": "立即预订 ¥599",
  "variant": "primary",
  "action": {
    "type": "callback",
    "event": "book_hotel",
    "payload": {
      "hotelId": "{{/hotels/0/id}}",
      "price": "{{/hotels/0/currentPrice}}"
    }
  }
}

示例：收藏 + 本地状态切换

Toggle Action {
  "id": "fav-btn",
  "component": "Button",
  "label": "♡ 收藏",
  "variant": "text",
  "action": {
    "type": "callback",
    "event": "toggle_favorite",
    "payload": { "hotelId": "{{/hotels/0/id}}" },
    "optimistic": {
      "path": "/hotels/0/isFavorited",
      "value": true
    }
  }
}

以"东京希尔顿 -35% 闪购卡片"为例，展示同一个 UI 如何被拆成五层独立描述。

Structure 骨架

[
  { "id": "deal-card", "component": "Card", "children": ["deal-img", "deal-body"] },
  { "id": "deal-img", "component": "Image", "aspectRatio": "16:9" },
  { "id": "deal-body", "component": "Column", "children": ["deal-badge", "deal-title", "deal-subtitle", "deal-price", "deal-cta"] },
  { "id": "deal-badge", "component": "Badge", "color": "error" },
  { "id": "deal-title", "component": "Text", "variant": "h2" },
  { "id": "deal-subtitle", "component": "Text", "variant": "caption" },
  { "id": "deal-price", "component": "PriceTag", "showOriginal": true },
  { "id": "deal-cta", "component": "Button", "variant": "primary" }
]

Data Model 数据

{
  "deal": {
    "image": "https://img.trip.com/hilton-shinjuku.jpg",
    "badge": "近30天最低",
    "title": "东京新宿希尔顿",
    "subtitle": "★4.7 · 含早 · 免费取消",
    "currentPrice": 599,
    "originalPrice": 890,
    "discount": "-35%",
    "ctaLabel": "立即预订",
    "hotelId": "htl_001"
  }
}

Catalog 约束（该卡片用到的类型）

// LLM 在生成上述结构时，只能从以下类型中选取：
Card, Image, Column, Badge, Text, PriceTag, Button

// 如果 LLM 输出了 "component": "VideoPlayer"
// → Validation 拒绝，因为 Catalog 中无此类型

Action 交互

// deal-cta 按钮的 action
{
  "type": "navigate",
  "url": "trip://hotel/detail/{{/deal/hotelId}}"
}

// 或回调 Agent 获取更多信息
{
  "type": "callback",
  "event": "view_deal_detail",
  "payload": { "hotelId": "{{/deal/hotelId}}" }
}

Theme 样式

// 该卡片继承全局 Theme，无需单独定义
// Card → borderRadius: theme.borderRadius.md (12px)
// Badge color="error" → theme.colors.error (#e74c3c)
// Text variant="h2" → theme.typography.h2 (18px/600)
// Button variant="primary" → bg: theme.colors.primary

训练数据格式建议

训练对 (Training Pair) 结构

Training Sample {
  "input": {
    "userIntent": "展示一个酒店闪购卡片，要有图片、折扣徽章、价格对比和预订按钮",
    "catalog": "[...可用组件列表...]",
    "dataContext": "[...当前可用数据字段...]"
  },
  "output": {
    "structure": "[...组件树 JSON...]",
    "bindings": "[...数据绑定...]",
    "actions": "[...交互定义...]"
  }
}

训练阶段建议

阶段训练内容评估标准
Phase 1: 结构生成需求 → 合法组件树所有 component 值在 Catalog 中；ID 唯一；children 引用有效
Phase 2: 数据绑定组件树 + 数据模型 → 正确 bindingJSON Pointer 路径存在于 Data Model 中
Phase 3: 交互声明组件树 → 合理 action 定义action type 合法；payload 字段可从 context 推导
Phase 4: 端到端自然语言需求 → 完整五层输出渲染后 UI 可用；交互链路完整

数据增强策略

结构变体

同一需求，生成多种布局（竖排/横排/网格），训练 LLM 理解布局选择。

数据变体

同一结构，绑定不同数据源（酒店/机票/门票），训练数据绑定泛化。

Catalog 变体

不同 Catalog 约束下生成 UI，训练 LLM 在边界内创造性组合。

负样本

包含"用了 Catalog 外组件"等违规输出，训练 LLM 识别并避免。

设计原则与约束

面向 LLM 的结构设计原则

  • 扁平优于嵌套 — 邻接表模型避免深层括号匹配，减少生成错误
  • 声明式优于命令式 — 描述"是什么"而非"怎么做"，安全且可验证
  • 流式友好 — 每个组件独立一行 JSONL，可逐条输出逐条渲染
  • 约束生成 — Catalog 枚举所有合法类型，validation 层自动拦截幻觉
  • 数据解耦 — 结构稳定时只更新数据，避免重新生成整棵树

常见反模式（训练中要避免）

  • 内联样式 — 组件中写死颜色/字号 → 应该通过 Theme token 间接引用
  • 硬编码数据 — 在结构层写死文案 → 应该通过 binding 引用 Data Model
  • 造新组件 — 输出 Catalog 不存在的 component → 应该用已有类型组合
  • 深层嵌套 — 7-8 层的树 → 应该拆分为多个扁平 surface
  • 混合关注点 — 在 action 中写业务逻辑 → 应该只声明事件名+负载

总结：一个组件 = 五张卡片

Structure（骨架）+ Data（血肉）+ Catalog（边界）+ Action（神经）+ Theme（皮肤）

训练时分层拆解，生成时按需组合。每一层都可以独立变更、独立验证、独立缓存。

页面级架构：从整体视角看 A2UI

单个组件是"砖"，页面级设计是"建筑图纸"。A2UI 在页面层面引入 Surface（画面） 概念——一个页面由多个 Surface 组成，每个 Surface 独立流式生成、独立更新。

页面 = 多 Surface 编排

┌─── Page ─────────────────────────────────────────────────────┐
│ │
│ ┌─ Surface: nav-bar ────────────────────────────────────┐ │
│ │ Logo · Search · UserAvatar (3 components) │ │
│ └────────────────────────────────────────────────────────┘ │
│ │
│ ┌─ Surface: flash-deals ────────────────────────────────┐ │
│ │ Countdown · HScroll[Card×3] (8-12 components) │ │
│ └────────────────────────────────────────────────────────┘ │
│ │
│ ┌─ Surface: discover ───────────────────────────────────┐ │
│ │ SectionTitle · Grid[DestCard×6](15-20 components) │ │
│ └────────────────────────────────────────────────────────┘ │
│ │
│ ┌─ Surface: for-you ────────────────────────────────────┐ │
│ │ SectionTitle · VList[DealCard×N](动态加载) │ │
│ └────────────────────────────────────────────────────────┘ │
│ │
└───────────────────────────────────────────────────────────────┘

每个 Surface 独立生成 · 独立更新 · 独立销毁
页面编排层决定 Surface 的顺序和可见条件

页面级分层

层级职责关键字段
Page Layout定义 Surface 排列顺序和加载策略surfaces[], loadStrategy
Surface一个独立 UI 区块，含自己的组件树和数据surfaceId, components[]
Component最小 UI 单元id, component, children
Slot/Repeat动态列表中的模板（一个模板 × N 数据）template, dataSource, maxItems

页面编排 Schema

Page Layout {
  "pageId": "deals-main",
  "layout": "scroll-vertical",
  "surfaces": [
    {
      "surfaceId": "nav-bar",
      "position": "sticky-top",
      "loadStrategy": "immediate"
    },
    {
      "surfaceId": "flash-deals",
      "position": "flow",
      "loadStrategy": "immediate",
      "visibility": { "condition": "hasActiveFlashDeals" }
    },
    {
      "surfaceId": "discover",
      "position": "flow",
      "loadStrategy": "immediate"
    },
    {
      "surfaceId": "for-you",
      "position": "flow",
      "loadStrategy": "lazy", // 滚动到视口时才加载
      "pagination": { "type": "infinite-scroll", "pageSize": 10 }
    }
  ]
}

组件数量约束与分屏策略

LLM 一次生成的 token 有限，组件数量直接影响生成质量和渲染性能。需要对每个 Surface 的组件数量设置上界。

组件数量分级

Surface 复杂度组件数上限适用场景Token 预估
轻量级3-8 个导航栏、工具栏、Toast 提示~200-500 tokens
标准级8-20 个卡片列表区块、表单、详情展示~500-1500 tokens
重量级20-40 个复杂仪表盘、多标签面板~1500-3000 tokens
超限 ⚠️> 40 个必须拆分为多 Surface→ 拆分

为什么需要上限？

LLM 生成质量

组件超过 30-40 个时，ID 引用错误率显著上升（括号不匹配、children 指向不存在的 ID）。

流式渲染性能

单 Surface 组件过多导致首次有意义绘制 (FMP) 延迟。拆分后首屏 Surface 可立即渲染。

增量更新效率

Surface 粒度越细，局部更新影响范围越小。整页一个 Surface = 任何变化都要 diff 全树。

训练数据标注

标注者在 20 个组件内可以完整验证结构正确性；超过后需要工具辅助。

Repeat/Template 模式：组件模板 × 数据列表

列表场景（10 个酒店卡片）不应该生成 10 × 7 = 70 个组件。应该定义 1 个模板 + 1 个数据数组，客户端负责重复渲染。

Repeat Pattern {
  "id": "deal-list",
  "component": "List",
  "direction": "vertical",
  "repeat": {
    "dataSource": "/deals", // Data Model 中的数组
    "maxItems": 10, // 单次最多渲染数
    "template": "deal-card-tpl" // 引用模板 Surface
  }
}

// 模板定义（只定义一次，复用 N 次）
{
  "templateId": "deal-card-tpl",
  "components": [
    { "id": "card", "component": "Card", "children": ["img", "body"] },
    { "id": "img", "component": "Image", "binding": "$.image" },
    { "id": "body", "component": "Column", "children": ["title", "price"] },
    { "id": "title", "component": "Text", "binding": "$.name" },
    { "id": "price", "component": "PriceTag", "bindings": { "current": "$.currentPrice" } }
  ]
}

// 效果：模板 5 个组件 × 10 条数据 = 客户端渲染 50 个实例
// 但 LLM 只需要生成 5 个组件定义 ✓
❌ 逐个生成

10 个卡片 × 7 组件 = 70 个组件

Token 爆炸 · ID 冲突 · 难以维护

✓ 模板 + 数据

1 个模板 (5组件) + 数据数组 = 5 个组件

高效 · 一致 · 数据驱动

LLM 生成策略：分步生成 vs 一次生成

一个完整页面不应该让 LLM 一次性生成全部 Surface。应该采用分步编排策略。

三种生成模式

模式描述适用场景组件上限
Single-shot一次生成整个 Surface简单区块（nav、banner）≤ 10
Sequential按 priority 逐个生成 Surface首屏加载（flash → discover）≤ 30/Surface
On-demand用户滚动/交互时才生成瀑布流（for-you）每批 ≤ 20

生成流程

Agent 收到请求："展示 Deals 首页"

Step 1: 生成 Page Layout（Surface 编排）
    → 输出：4 个 Surface 定义 + 加载策略
    → token cost: ~200

Step 2: 并行生成 immediate Surface
    → nav-bar: 4 组件 (~150 tokens)
    → flash-deals: 6 组件模板 (~400 tokens)
    → discover: 5 组件模板 (~350 tokens)

Step 3: 用户滚动到 for-you 区域时
    → for-you: 7 组件模板 (~500 tokens)
    → 首批 10 条数据绑定

Step 4: 继续滚动时
    → 只发 updateDataModel（追加数据）
    → 复用已有模板，零组件生成

总计 LLM 生成：~22 个组件 + ~1600 tokens
客户端渲染：~129 个组件实例

约束规则与验证清单

页面级硬约束（用于训练 validation）

约束项上限违规处理
单 Surface 组件数30拒绝，要求拆分
页面 Surface 总数8拒绝，要求合并
LLM 单次生成组件总量50截断 + 分步提示
组件嵌套深度4 层拒绝，要求扁平化
单 repeat 列表项数20强制分页
ID 命名长度32 字符截断警告
单组件 binding 数10警告，建议拆分
Action payload 字段数8警告，建议精简

训练验证清单

结构验证

  • 所有 children 引用的 ID 必须在同一 Surface 内存在
  • 无环引用（A → B → A）
  • 有且只有一个 root 组件（无父引用）
  • 所有 component 值在 Catalog 中存在
  • ID 全局唯一（跨 Surface 也不能重复）

数据验证

  • 所有 binding 路径在 Data Model 中可达
  • repeat 的 dataSource 指向数组类型
  • 模板内 binding 使用 $. 相对路径
  • 无未绑定的动态字段（text 或 binding 二选一）

性能验证

  • 首屏 Surface（immediate）组件总数 ≤ 40
  • lazy Surface 不在首次生成中出现
  • 无单 Surface 超过 30 组件
  • 嵌套深度 ≤ 4（root 算第 1 层）

页面级分层对训练的意义

训练阶段输入输出评估标准
Phase 1: 页面规划需求描述 + CatalogPage Layout（Surface 编排）Surface 划分合理；加载策略正确
Phase 2: Surface 生成单 Surface 需求 + 数据模型组件树 + binding组件数 ≤ 上限；结构验证通过
Phase 3: 模板设计列表场景描述template + repeat 定义用模板复用而非逐个生成
Phase 4: 增量更新状态变化描述updateDataModel / updateComponents最小变更集；不重建整树
Phase 5: 端到端完整页面需求分步生成完整页面遵守所有约束；渲染后可用

核心公式

一个页面 = N 个 Surface × 每 Surface M 个模板组件 × Data 驱动 K 个实例

LLM 生成量 = N × M（通常 20-50）
实际渲染量 = Σ(Mi × Ki)（可达数百）

训练目标：让 LLM 学会"用最少的模板组件覆盖最多的 UI 实例"。

结构不变、风格可变的关键设计

Structure 层无视觉信息

组件树只描述"有什么"和"层级关系"，不包含颜色、间距、字号。同一 Structure 在中国和美国渲染完全不同的视觉。

Theme Token 而非硬编码值

使用 color.promo 而非 #FF0000。Token 在不同 Region Profile 中解析为不同值。

条件组件可见性

某些组件在特定区域才显示（如中东显示里程积分、中国显示券面、欧美显示评分细分）。通过 visibility.condition 控制。

布局方向自适应

RTL 区域自动镜像 Row 组件的 children 顺序，无需 LLM 重新生成结构。客户端 renderer 处理。
